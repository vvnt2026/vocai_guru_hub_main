# Vercel Deployment Audit: VocAI Guru Hub

This document provides a comprehensive analysis of the project structure and configuration for **VocAI Guru Hub**, a React application built with the TanStack Start framework, and details the steps required to deploy it successfully on Vercel.

---

## 1. Detected Tech Stack

* **Meta-Framework:** [TanStack Start (v1)](https://tanstack.com/router/latest/docs/framework/react/start/overview) — an SSR/full-stack framework built on top of TanStack Router, Vite, and Nitro.
* **Core Library:** React 19
* **Routing:** TanStack Router (v1)
* **Styling:** Tailwind CSS v4
* **Build Tooling:** Vite v7 + Nitro Build Engine
* **Server Runtime:** Node.js (via Nitro presets)
* **Package Manager:** Bun (preferred, based on `bun.lock` / `bunfig.toml` in the project root; npm is supported as a fallback)

---

## 2. Key Issues Identified with Static Deployment

The original project configuration assumed a static frontend SPA deployment (via a custom `postbuild.mjs` and static rewrite rules in `vercel.json`). However, several factors prevent a purely static deployment:

1. **No Static HTML Entry Point:**
   During a standard `vite build` without Nitro, TanStack Start outputs client bundles (`dist/client/assets`) and server bundles (`dist/server`), but **does not output an `index.html` file**. Since there is no entry-point file in the `dist` output, a static deployment fails immediately with routing/404 errors.
   
2. **Server-Side Functions (`createServerFn`):**
   The application contains server functions in `src/lib/ai.functions.ts` (e.g., `askMentor`, `generateLessonPlan`, `recommendTrades`) that handle backend tasks.
   * **Security:** These functions securely retrieve the `LOVABLE_API_KEY` using `process.env.LOVABLE_API_KEY` to query the Lovable AI gateway.
   * **Execution Requirement:** If deployed on a purely static platform, there is no server-side execution context. When the browser makes a request to execute a server function (via `/_server`), Vercel's static router will fail with a 404 (or serve `index.html` fallback), rendering the AI mentor, parent assistant, and recommendations completely non-functional.
   
3. **Conflicting static configuration in `vercel.json`:**
   The original `vercel.json` forced `outputDirectory: "dist"` and redirected all routing to `/index.html`. This tells Vercel to look only for static files, bypassing the Nitro serverless backend.

## 3. Recommended & Implemented Changes

To ensure both secure server-side execution and fast static asset delivery, we have configured the project to use **Vercel's Build Output API (v3)** via Nitro's Vercel preset.

### A. Modified Configuration Files

We have already modified the configuration files in your repository to support Vercel SSR deployment:

#### 1. `vite.config.ts`
Added `nitro: { preset: 'vercel' }` to the Vite configuration:
```typescript
import { defineConfig } from "@lovable.dev/vite-tanstack-config";

export default defineConfig({
  tanstackStart: {
    server: { entry: "server" },
  },
  nitro: {
    preset: "vercel", // Forces Nitro to compile to .vercel/output for Vercel deployment
  },
});
```
* **Reason:** This compiles the application into the Vercel Build Output format. It splits static assets into `.vercel/output/static` (served via CDN) and packages server functions into a serverless function under `.vercel/output/functions/__server.func` (run on demand).

#### 2. `vercel.json`
Simplified to avoid static file conflicts:
```json
{
  "buildCommand": "vite build"
}
```
* **Reason:** Removing the static `outputDirectory` and `rewrites` allows Vercel to automatically detect the `.vercel/output` folder generated by the build and deploy it as a hybrid serverless/static SSR app.

## 4. Potential Risks & Mitigation

| Potential Risk | Impact | Mitigation / Solution |
| :--- | :--- | :--- |
| **Missing Environment Variables** | AI features fail to fetch responses and return API errors. | Ensure `LOVABLE_API_KEY` is added to the Environment Variables section in the Vercel project settings dashboard. |
| **Node.js Runtime Mismatch** | The build compiles for Node.js v22. If Vercel defaults to an older runtime, bundling errors might occur. | Vercel automatically detects the Node.js version. It is recommended to keep the Vercel Node runtime version at `22.x` or `20.x` in your Vercel Project Settings. |
| **Client-side API Key Exposure** | None (Mitigated). | By using `createServerFn`, all API calls to the Lovable AI gateway are kept on Vercel's secure serverless side. The key is never sent to the browser. |

## 5. Step-by-Step Vercel Deployment Instructions

Follow these instructions to deploy your website on Vercel:

### Step 1: Push Project to GitHub / GitLab / Bitbucket
Ensure your local project with the updated configuration files (`vite.config.ts` and `vercel.json`) is pushed to your Git repository:
```bash
git add vite.config.ts vercel.json vercel-deployment-audit.md
git commit -m "Configure hybrid Vercel deployment for TanStack Start"
git push origin main
```

### Step 2: Import Project to Vercel
1. Log in to the [Vercel Dashboard](https://vercel.com).
2. Click **Add New** -> **Project**.
3. Import your Git repository (`vocai-guru-hub`).

### Step 3: Configure Project Settings on Vercel
On the configuration screen:
* **Framework Preset:** Leave as **Other** (Vercel will auto-detect the `.vercel/output` folder generated by the build).
* **Build Command:** Keep as default (`npm run build` or `bun run build`).
* **Output Directory:** Keep blank / default (Vercel will read `.vercel/output` automatically).

### Step 4: Add Environment Variables (Crucial)
Under the **Environment Variables** section, add your secret API key:
* **Key:** `LOVABLE_API_KEY`
* **Value:** *(Paste your Lovable API Key)*

Click **Add**.

### Step 5: Deploy
Click **Deploy**. Vercel will run the build, package the React client assets on the CDN, deploy the server functions, and provide a live production URL!
